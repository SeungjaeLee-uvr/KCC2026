System Overview

객체 단위 저장소와
XR 내부 제어 UI로 구성

Shared Virtual Object

position, rotation, scale, color, interactable, sensitivity를 객체 상태로 캡처

In-world Control UI

commit, branch, PR, merge, checkout을 XR 공간에서 실행

JSON Repository

commit, branch, PR, conflict 정보를 기기 내부에 저장

Repository Model

전체 장면 대신,
의미 있는 객체 상태만 버전 관리

RepositoryData

repositoryName, activeBranchName, commits, branches, pullRequests 보존

CommitNode

parentCommitId, branchName, object state, changedAttributes 저장

ObjectState

위치, 회전, 크기, 색상 등 저수준 속성 캡처와 재적용

핵심 효과: 전체 scene snapshot보다 가벼움, 사용자가 말할 수 있는 수정 단위와 더 가까움

Commit & Branch

기존 진행을 깨지 않고,
과거 상태에서 새 대안 생성

Branch lanes

main branch와 sub branch를 공간적으로 분리해 lineage 확인

Detached commit

과거 commit 수정 시 새 branch 자동 생성, 기존 진행 덮어쓰기 방지

Pull Request & Merge

수정안은 바로 반영하지 않고,
검토 가능한 PR로 통합

1Branch Head

대안 객체 상태 준비

2Pull Request

main head와 branch head의 변경 속성 요약

3Three-way Diff

base, main, branch 비교로 충돌 여부 판단

4Merge Commit

충돌 없음 또는 해결 시 main에 새 commit 생성

Conflict Resolution

충돌은 텍스트 목록이 아니라,
XR 공간의 미니어처 비교로 제시

Main vs PR Head

두 revision을 side-by-side miniature로 비교, 변경 결과를 공간적으로 확인

Attribute choices

Rotation, Scale 등 충돌 속성별 Main 또는 Branch 값 선택

Demo Plan

데모 핵심은 하나의 객체가
이력을 갖는 과정 제시

  1. 초기 상태: main branch와 initial commit 생성
  2. 수정과 commit: 객체 이동 또는 색상 변경 후 현재 상태 commit
  3. branch 실험: branch 생성 후 다른 수정안 작성
  4. PR 생성: main과 branch의 변경 속성 비교
  5. conflict review: 미니어처와 텍스트 요약 기반 값 선택
  6. merge: 해결 상태를 main의 새 merge commit으로 저장
